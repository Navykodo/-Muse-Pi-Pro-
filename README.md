# 基于 K1 Muse Pi Pro 与龙虾智能体的家居助手

本仓库是比赛作品“基于进迭时空 K1 Muse Pi Pro 与龙虾智能体的家居助手”的源码整理与关键实现归档，用于展示系统的软件结构、硬件接口封装、智能体调用规则、Android 移动端上位机和底层设备控制实现。它不是一键部署包，也不包含现场运行所需的私有配置、模型权重、日志、构建产物或设备镜像。

项目面向智能家居与室内服务场景，目标是将大模型智能体从纯软件对话扩展到真实物理环境中。系统整合语音唤醒与识别、声源定位与播报、摄像头视觉理解、温湿度检测、小车运动与导航、智能家居控制以及异常告警等能力，使智能体能够理解用户的自然语言指令，并通过受控接口调度真实硬件完成任务。

## 作品概述

用户可以通过语音、Web 页面或 Android App 发出指令，例如“去门外走廊看一下”“巡逻一圈”“看看室内有什么异常吗”。系统会将这些模糊自然语言任务交给龙虾智能体理解，再依据本地 skill 规则拆分为可执行的子任务，按需调用小车移动、摄像头拍摄、温湿度读取、视觉识别、语音播报、智能插座控制和告警推送等工具。

本作品强调长任务的自主规划、执行与修正。对于巡逻、导航后观察、异常复核等任务，系统不把“命令已发送”视为完成，而是持续读取硬件状态和工具返回，根据过程结果决定是否等待、重试、复核、补光或停止，直到形成可向用户反馈的结果。

## 应用场景

- 智能家居控制：查询环境、控制家电、通过自然语言触发硬件动作。
- 家庭安全巡检：识别烟雾、火焰、积水、裸露线缆、人员倒地等风险。
- 室内服务机器人：支持门口、窗户、走廊等语义地点导航和固定路线巡逻。
- 适老化看护：通过巡逻、视觉观察和移动端强提醒降低异常漏报风险。
- 教学与实践：展示大模型智能体如何连接传感器、摄像头、小车和移动端应用。

## 系统架构

系统采用“智能体 + 本地硬件能力 + 移动端上位机”的分层架构。

```text
用户交互层
  -> 语音唤醒 / Web 调试页面 / Android App
智能体决策层
  -> 龙虾智能体 / 大模型 API / skill 规则 / 任务状态控制
硬件中控层
  -> Hardware API / Tool Router / 状态锁 / 结构化返回
设备执行层
  -> 摄像头 / 温湿度 / 小车底盘 / ROS 2 导航 / 智能家居 / 告警服务
移动端上位机层
  -> 视频查看 / 摇杆遥控 / 状态查询 / 告警强提醒
```

大模型不直接操作摄像头、小车底盘、ROS 2 节点或底层端口，而是通过 Hardware API 调用受控工具。这样既保留自然语言理解和长任务规划能力，也避免模型直接接触底层硬件带来的误调用、参数错误和并发冲突。

## 核心功能

### 自然语言任务闭环

系统支持语音、Web 和 App 多入口交互。用户无需记忆固定命令，只需描述目标，智能体会根据 skill 规则规划工具调用顺序。执行过程中，系统记录工具名、参数、返回结果和耗时，便于比赛展示、问题排查和复盘。

### 语音交互与 ASR

语音链路基于麦克风阵列完成唤醒、声源方向解析和音频采集，并通过本地 ASR 服务将语音转换为文本后发送给智能体。系统在唤醒后会打断正在播放的 TTS，降低自身播报被再次录入的概率。

### 硬件中控与工具路由

Hardware API 对外提供统一 `/tool` 接口。不同底层能力被抽象为“工具名 + 参数 + 结构化结果”，返回字段统一为 `ok`、`data`、`error`。运动、导航、拍照、播报和告警等可能冲突的任务通过锁、状态字段和超时控制降低并发风险。

### 移动导航与巡逻

ROS 2/Nav2 服务用于室内定位、导航和语义地点移动。系统区分“导航目标已提交”和“小车实际到达”，通过状态轮询确认 `nav_state`、`nav_done` 和 `nav_ok`，避免把接口接收成功误判为任务完成。

### 异常识别与二次确认

系统可通过摄像头视觉理解识别人员、家具、门窗、烟雾、火焰、积水、裸露线缆、人员倒地等室内状态。针对误判风险，异常检测加入二次确认机制；若环境过暗，可联动灯光补光观察，并在任务结束后恢复原状态。

### Android 移动端上位机

Android App 支持硬件状态查询、摄像头视频查看、小车虚拟摇杆遥控和告警接收。危险告警到达后，App 会尝试拉起前台，显示应用内弹窗，同时触发系统通知、震动和警报音，直到用户在 App 内确认处理。

## 主要创新点

1. 将大模型智能体从文本对话扩展到真实家居场景，使其具备调用传感器、摄像头、小车和家居设备的物理执行能力。
2. 构建“感知、理解、执行、反馈”的闭环流程，支持从模糊自然语言指令到多步骤硬件执行的自主规划与修正。
3. 使用 Hardware API 和 skill 规则作为智能体与硬件之间的安全边界，降低真实设备误调用风险。
4. 将异常识别、二次确认、移动端推送和 App 强提醒连接为完整安全告警链路。
5. 支持语音、Web 和 Android App 多入口交互，兼顾自然使用、开发调试和移动端响应。
