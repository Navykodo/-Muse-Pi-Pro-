# 本地语音提醒 / cron 技能

当用户表达以下意图时使用本技能：

- “1 分钟后提醒我喝水”
- “十分钟后叫我吃药”
- “下午三点提醒我开会”
- “明天早上 8 点叫我起床”
- “每天早上 9 点提醒我打卡”
- “每隔 30 分钟提醒我站起来活动”
- “设置一个闹钟 / 倒计时 / 提醒”

你的目标不是口头承诺，而是**必须创建 ZeroClaw cron 任务**。创建成功后再简短确认。

## 硬性规则，必须遵守

1. **不要把北京时间直接写进 `Z` 时间**：`Z` 表示 UTC，不是中国时间。
2. **一次性 `at` 任务必须先按 `Asia/Shanghai` 理解用户时间，再换算成 UTC ISO8601**。
3. **相对时间优先用当前真实时间计算**：例如当前本地时间 `14:58:24`，用户说“3 分钟后”，目标本地时间是 `15:01:24`。
4. **中国时间转 UTC 必须减 8 小时**：`Asia/Shanghai 15:01:24` 应写成 `07:01:24Z`，绝不能写成 `15:01:24Z`。
5. **创建前自检**：如果用户说“N 分钟后”，`at` 换算回本地时间后，必须约等于当前本地时间加 N 分钟。

## 关键通道

ZeroClaw cron 到点后，直接调用 Hardware API 的 `speak_text` tool 播报：

```text
POST http://127.0.0.1:8765/tool
Content-Type: application/json
Body: {"tool":"speak_text","args":{"text":"要播报的提醒内容"}}
```

不要再使用旧的 `http://127.0.0.1:8787/cron_event`。

## 必须使用的工具

优先使用 ZeroClaw 内置工具：

```text
cron_add
```

不要只回答“好的我会提醒你”。必须确认 `cron_add` 成功。

## 一次性提醒

用户说“几分钟后 / 几小时后 / 今天几点 / 明天几点 / 某个具体时间提醒我”时，创建一次性任务：

```json
{
  "schedule": {
    "kind": "at",
    "at": "<UTC ISO8601 时间>"
  },
  "job_type": "shell",
  "command": "<curl POST Hardware API speak_text 的命令>",
  "delete_after_run": true
}
```

要求：

- `schedule.kind` 必须是 `at`。
- `at` 必须是 UTC ISO8601，例如 `2026-05-20T12:01:00Z`。
- `at` 末尾带 `Z` 时就是 UTC 时间，禁止把 `Asia/Shanghai` 的本地时钟数值直接填进去。
- `job_type` 必须是 `shell`。
- `delete_after_run` 必须是 `true`。
- 到点后播报的文字必须放在 `args.text` 字段。

### 一次性提醒 curl 模板

```bash
curl -fsS -X POST 'http://127.0.0.1:8765/tool' \
  -H 'Content-Type: application/json' \
  --data-binary '{"tool":"speak_text","args":{"text":"该喝水了"}}'
```

## 重复提醒

用户说“每天 / 每周 / 每隔一段时间 / 每小时”等重复提醒时，创建重复任务，且不要自动删除。

### 每隔 N 分钟/小时

使用 `every`：

```json
{
  "schedule": {
    "kind": "every",
    "every_ms": 1800000
  },
  "job_type": "shell",
  "command": "<curl POST Hardware API speak_text 的命令>",
  "delete_after_run": false
}
```

### 每天/每周固定时间

使用 `cron`，优先带中国时区：

```json
{
  "schedule": {
    "kind": "cron",
    "expr": "0 9 * * *",
    "tz": "Asia/Shanghai"
  },
  "job_type": "shell",
  "command": "<curl POST Hardware API speak_text 的命令>",
  "delete_after_run": false
}
```

常见 cron：

| 用户说法 | expr | tz |
|---|---|---|
| 每天早上 9 点 | `0 9 * * *` | `Asia/Shanghai` |
| 每天晚上 8 点 | `0 20 * * *` | `Asia/Shanghai` |
| 每周一上午 9 点 | `0 9 * * 1` | `Asia/Shanghai` |
| 工作日早上 8 点 | `0 8 * * 1-5` | `Asia/Shanghai` |

## 时间理解规则

- 相对时间：“1 分钟后”“半小时后”“2 小时后”按当前时间向后计算，然后转成 UTC ISO8601。
- 本地口语时间默认按 `Asia/Shanghai` 理解。
- `Asia/Shanghai` 是 UTC+8；换成 `Z` 时间必须减 8 小时。
- “今天下午三点”如果已经过了，应询问用户确认，除非上下文明确是明天。
- “明天早上八点”按下一天 `Asia/Shanghai 08:00`，再转成 UTC。
- 时间不明确时先追问，不要创建错误任务。

### UTC 换算强制示例

| 当前本地时间 Asia/Shanghai | 用户请求 | 目标本地时间 Asia/Shanghai | cron_add 的 `at` 必须填写 |
|---|---|---|---|
| `2026-05-25 14:58:24` | `3 分钟后提醒我` | `2026-05-25 15:01:24` | `2026-05-25T07:01:24Z` |
| `2026-05-25 08:00:00` | `10 分钟后提醒我` | `2026-05-25 08:10:00` | `2026-05-25T00:10:00Z` |
| `2026-05-25 23:59:00` | `2 分钟后提醒我` | `2026-05-26 00:01:00` | `2026-05-25T16:01:00Z` |
| `2026-05-25 14:00:00` | `下午三点提醒我` | `2026-05-25 15:00:00` | `2026-05-25T07:00:00Z` |

错误示例：

```json
{
  "schedule": {
    "kind": "at",
    "at": "2026-05-25T15:01:24Z"
  }
}
```

如果当前中国时间是 `2026-05-25 14:58:24`，用户说“3分钟后”，上面这个写法是错的，因为它会在中国时间 `23:01:24` 触发。正确应是 `2026-05-25T07:01:24Z`。

## 提醒文本生成规则

从用户请求中提取实际要播报的内容。

示例：

| 用户请求 | text |
|---|---|
| 1 分钟后提醒我喝水 | `该喝水了` |
| 10 分钟后叫我吃药 | `该吃药了` |
| 下午 3 点提醒我开会 | `该开会了` |
| 明天早上 8 点叫我起床 | `该起床了` |
| 每隔 30 分钟提醒我站起来活动 | `该站起来活动一下了` |

## 创建前自检清单

调用 `cron_add` 前必须在内部完成以下检查：

1. 用户是一次性提醒还是重复提醒？
2. 一次性提醒的本地目标时间是多少？
3. 本地目标时间转成 UTC 后是多少？
4. `at` 是否以 `Z` 结尾并且已经减去 8 小时？
5. `command` 是否调用 `http://127.0.0.1:8765/tool`，而不是旧的 `8787/cron_event`？

## 完成后的回复

创建成功后，用中文简短确认：

- “好的，1 分钟后提醒你喝水。”
- “好的，明天早上 8 点叫你起床。”
- “好的，我会每天早上 9 点提醒你打卡。”

不要输出 curl 细节，除非用户要求。失败时说明失败原因。
