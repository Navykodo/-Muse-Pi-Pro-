# 异常与报警技能

用于处理“异常判定”和“发送 App 告警”。视觉类可复核异常需要先二次确认；已经确认的异常、传感器明确异常、明显明火高危场景和用户测试告警必须发送告警，不能只在回复里描述异常。

## 触发场景

- 用户要求“发送告警 / 推送报警 / 通知 App”。
- 用户要求发送某类测试告警，例如火焰、烟雾、积水、入侵者、障碍物、门窗异常。
- 视觉、哨兵、巡逻、导航观察或环境观察发现 `confirmed_abnormal`。
- 硬件传感器明确上报异常。
- 视觉观察到明显明火、正在燃烧、爆炸等不能等待的高危场景。
- 其它 skill 明确要求“发现异常后调用告警 skill”。

## 什么是异常

以下情况都算异常。视觉首次发现其中的可复核异常时，先按 `camera-snap-tools` 二次确认；确认后必须告警：

- 火焰、明火、疑似起火、烧焦烟雾。
- 烟雾、浓雾、异常雾气、明显空气异常。
- 积水、大面积水渍、漏水、地面湿滑风险。
- 裸露线缆、电线散落、电气设备异常、疑似短路火花。
- 悬空边缘、台阶边缘、坑洞、可能导致小车跌落的位置。
- 明显贴近车体、阻挡运动或可能碰撞的障碍物。
- 门窗异常打开、破损、遮挡异常、门外/走廊出现异常物体。
- 入侵者、陌生人异常进入、人员异常靠近设备。
- 人员摔倒、倒地、躺在地上、趴在地上、长时间异常静止。
- 视觉描述里明确出现“异常”“危险”“需要检查”“疑似事故”“不安全”等判断。

以下情况通常不算异常，除非视觉结果明确说有危险：

- 普通家具、桌椅、柜子、墙面、窗帘、正常堆放物。
- 人员正常坐着、站着、走动，且没有异常姿态或危险动作。
- 地面不可见、光线偏暗、画面有普通遮挡；这些只能说“不确定”，不能直接告警。

## 检测到异常后的强制动作

检测到上面任一明确异常时：

1. 如果是首次视觉结果且不是明显明火/正在燃烧等高危场景，先按 `camera-snap-tools` 做二次确认。
2. 二次确认得到 `confirmed_abnormal`、硬件传感器明确异常、明显明火高危场景或用户明确要求测试告警时，必须发送告警。
3. 如果二次确认未复现同类异常，标记为 `unconfirmed_suspect`，不要发送告警，只在最终回复里简短说明。
4. 如果当前还有小车运动或巡逻任务，应先按对应运动/巡逻 skill 停止小车，再发送告警。
5. 已确认异常后不要继续执行后续巡逻或观察步骤。
6. 不要把 `confirmed_abnormal` 只写在最终回复里而不发告警。
7. 如果告警发送失败，必须报告真实失败原因，不能伪造已发送。

## 服务地址

当前告警 TCP 地址由部署环境配置：

```text
ALERT_TCP_HOST:ALERT_TCP_PORT
```

这是原始 TCP 协议，不是 HTTP。不能用浏览器、`curl` 或 `http_request` 测试这个端口。

服务端是广播转发模型：所有连接到同一端口的客户端在同一个广播组里，任意客户端发送的数据会原样转发给其它当前在线客户端。App 必须已经连接在线，发送时才会收到。服务端不缓存历史消息。

## 推荐脚本

优先使用部署环境中配置好的告警发送脚本：

```text
send_alert.py LEVEL EVENT
```

脚本只接受两个位置参数：

- `LEVEL`：只能是 `info` / `warning` / `danger`
- `EVENT`：告警事件文本，也就是 App 显示的内容

示例：

```bash
send_alert.py danger "检测到火焰，请立即检查"
send_alert.py warning "前方检测到障碍物"
send_alert.py info "哨兵模式已开启"
```

不要再使用旧参数：

```text
--level --code --host --port --timeout --repeat --interval --no-ts --extra --dry-run
```

## 数据格式

每条告警是一行 UTF-8 JSON，并且必须以换行符 `\n` 结尾。

当前固定格式：

```json
{"type":"alert","level":"danger","message":"检测到火焰，请立即检查"}
```

字段：

- `type`：必填，固定为 `"alert"`。
- `level`：必填，只能是 `"info"`、`"warning"`、`"danger"`。
- `message`：必填，人类可读的告警事件文本。

当前格式不要添加：

- `code`
- `ts`
- `extra`
- 其它自定义字段

## 换行规则

- JSON 中间不能有真实换行。
- 只能在最后有一个换行符，用来通知 App 一条消息结束。
- App 端按行读取，真实换行会把一条 JSON 拆成多条无效数据。
- 如果告警文本来自多行内容，先压缩成单行再发送。

中文消息本身没问题。推荐使用 `send_alert.py`，它会自动生成单行 JSON。

## 等级建议

- 火焰、烟雾、积水、疑似起火：默认 `danger`。
- 人员摔倒、倒地、躺在地上、趴在地上：至少 `warning`；疑似受伤、无法移动或伴随其它危险时用 `danger`。
- 裸露线缆、悬空边缘、电气异常：默认 `warning`；明确危险时用 `danger`。
- 入侵者、人员异常靠近、门窗异常：默认 `warning`；明确危险时用 `danger`。
- 障碍物：默认 `warning`；贴近车体、阻挡运动或可能碰撞时用 `danger`。
- 普通状态提示：默认 `info`。

## 成功和失败判断

可以说“告警已发送”的条件：

1. `send_alert.py` 退出码为 0。
2. 输出类似 `sent: danger -> 检测到火焰，请立即检查`。

失败时报告真实错误，例如连接拒绝、超时、BrokenPipe、ConnectionRefused 或脚本参数错误。

注意：发送成功只代表 TCP 数据已发送到广播服务，不代表 App 一定已展示。App 必须处于“告警 TCP 已连接”状态。

## 没有脚本或 shell 时

如果当前工具集没有 shell，也没有 TCP/socket 发送能力，必须直接说明无法发送。

不要用这些方式兜底：

- `http_request`
- browser
- curl
- 访问网页

## 回复规则

回复保持简短，包含：

- `level`
- `message`
- 成功或失败原因

不要再输出或要求 `code` / `ts`。不要输出过长原始 JSON，不展示内部思考过程。
